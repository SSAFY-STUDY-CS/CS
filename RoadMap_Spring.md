### 1주차. Spring Boot의 탄생 배경

- **핵심 질문**: 왜 Spring은 Boot로 진화했는가?
- **핵심 포인트**
    - 설정의 비대화(Configuration Hell)와 XML 방식의 한계
    - 부트의 자동 설정(Auto-configuration)이 문제를 해결하는 방식
    - '설정의 제거'가 아닌 '설정의 추상화와 은닉' 이해

### 2주차. Bean Container & ApplicationContext

- **핵심 질문**: Spring은 객체를 어떻게, 그리고 언제 관리하는가?
- **핵심 포인트**
    - IoC(제어의 역전) 컨테이너가 객체 생명주기를 책임지는 이유
    - BeanDefinition을 통한 객체 생성 정보의 추상화
    - BeanFactory와 ApplicationContext의 계층적 역할

### 3주차. DI(Dependency Injection)의 본질

- **핵심 질문**: new를 사용한 객체 생성이 유지보수에 끼치는 영향은?
- **핵심 포인트**
    - 의존 관계 주입을 통한 객체 간 결합도 해소
    - 인터페이스 기반 설계와 구현체 교체의 유연성
    - DI가 단위 테스트(Mocking) 가능성에 기여하는 지점

### 4주차. Spring MVC 내부 동작 구조

- **핵심 질문**: HTTP 요청은 어떤 과정을 거쳐 Controller에 도달하는가?
- **핵심 포인트**
    - 서블릿(Servlet) 컨테이너와 DispatcherServlet의 관계
    - Front Controller 패턴을 통한 공통 관심사 처리
    - 요청 매핑, 핸들러 어댑터, 뷰 리졸버의 협력 모델

### 5주차. JPA를 쓰는 이유와 위험성

- **핵심 질문**: ORM은 개발을 편하게 만들지만, 왜 위험할 수 있는가?
- **핵심 포인트**
    - 반복적인 리소스 관리의 위임(템플릿 사상의 적용)
    - 영속성 컨텍스트가 추상화하는 데이터베이스 상태
    - 개발자가 제어 권한을 어디까지 인지하고 있어야 하는가에 대한 고찰

### 6주차. @Transactional과 AOP

- **핵심 질문**: 트랜잭션 코드는 어떻게 비즈니스 로직에 개입하는가?
- **핵심 포인트**
    - 프록시(Proxy) 객체를 이용한 횡단 관심사(AOP) 분리
    - 선언적 트랜잭션의 동작 원리와 프록시의 한계
    - Self-invocation(자기 호출) 시 발생하는 구조적 문제

### 7주차. Spring의 동시성 & 성능

- **핵심 질문**: 싱글톤 Bean은 멀티스레드 환경에서 왜 안전하지 않은가?
- **핵심 포인트**
    - 무상태(Stateless) 설계가 싱글톤 환경에서 필수적인 이유
    - ThreadLocal의 사용 배경과 메모리 누수 위험성
    - 컨테이너가 보장하는 영역과 개발자가 책임져야 하는 영역의 경계

### 8주차. Spring 아키텍처 설계 관점

- **핵심 질문**: '스프링답게 설계한다'는 말의 실체는 무엇인가?
- **핵심 포인트**
    - 계층 분리(Layered Architecture)의 진짜 목적
    - 서비스 계층 비대화(Fat Service) 문제와 책임의 재배치
    - PSA(Portable Service Abstraction)를 통한 기술 종속성 제거
