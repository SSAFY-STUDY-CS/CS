# [보충5.2] Semaphore 동작 과정

## 1. 세마포어(Semaphore) 동작 원리

세마포어는 음이 아닌 정수형 변수 **`S`**와 두 가지 원자적 연산 **`P()`**, **`V()`**를 통해 자원을 관리합니다.

- **변수 S (세마포어 값):** 현재 사용 가능한 자원의 개수를 의미합니다.
- **초기화 연산:** S 변수에 초기값을 부여하는 연산
- **P(S) 연산 (Proberen, '테스트'):** 자원을 확보하는 과정입니다. `S`가 0보다 크면 `S`를 감소시키고 진입하며, 0이면 자원이 생길 때까지 대기합니다.
    
    ![image.png](image%2011.png)
    
- **V(S) 연산 (Verhogen, '증가'):** 자원을 반납하는 과정입니다. `S`를 증가시키고, 대기 중인 프로세스가 있다면 하나를 깨웁니다.
    
    ![image.png](image%2012.png)
    
- 임의의 **S 변수** 하나에 **Ready queue** 하나가 할당됨

## 2. 자세한 동작 과정 (Block-Wakeup 모델)

현대 운영체제의 세마포어는 CPU 낭비를 막기 위해 **Busy Waiting이 없는 Block-Wakeup** 방식으로 동작합니다.

1. **P(S) 호출:** * `S.value`를 1 감소시킵니다.
    - 만약 `S.value < 0`이라면, 자원이 없다는 뜻이므로 현재 프로세스를 **세마포어 대기 큐(Wait Queue)**에 넣고 **Block(Sleep)** 상태로 전환합니다.
2. **임계 영역(CS) 수행:** 자원을 획득한 프로세스만 안전하게 코드를 실행합니다.
3. **V(S) 호출:** * `S.value`를 1 증가시킵니다.
    - 만약 `S.value <= 0`이라면, 대기 큐에 기다리는 프로세스가 있다는 뜻입니다.
    - 대기 큐에서 프로세스 하나를 꺼내 **Ready Queue**로 옮겨 **Wakeup** 시킵니다.

## 3. 각각의 동기화 문제 해결 과정

### **3.1** 상호배제 문제 (Mutual Exclusion)

![image.png](image%2013.png)

- **목적:** 한 번에 하나의 프로세스만 임계 영역에 들어가도록 보장.
- **설정:** 세마포어 `mutex`를 **1**로 초기화합니다.
- **과정:**
    
    `P(mutex); // 진입 전 검사 (S: 1 -> 0)
    // Critical Section
    V(mutex); // 퇴출 후 알림 (S: 0 -> 1)`
    

### **3.2** 프로세스 동기화 문제 (Process Synchronization)

![image.png](image%2014.png)

- **목적:** 프로세스 간의 실행 순서를 강제함 (예: P1이 끝난 후 P2가 실행되어야 함).
- **설정:** 세마포어 `sync`를 **0**으로 초기화합니다.
- **과정:**
    - **Pi:** `P(sync); 코드 실행;` (`sync`가 0이므로 Pj가 신호를 주기 전까지 대기)
    - **Pj:** `코드 실행; V(sync);` (작업 완료 신호를 보냄)

### **3.2** 생산자-소비자 문제 (Producer-Consumer Problem)

![image.png](image%2015.png)

- **목적:** 버퍼가 가득 찼을 때 생산자 대기, 버퍼가 비었을 때 소비자 대기.
- **설정:** * `mutex = 1` (버퍼 접근 보호)
    - `empty = N` (비어있는 칸의 개수)
    - `full = 0` (차있는 칸의 개수)
- **과정 (생산자):**`P(empty); P(mutex); 버퍼에 추가; V(mutex); V(full);`
- **과정 (소비자):**`P(full); P(mutex); 버퍼에서 제거; V(mutex); V(empty);`

![image.png](image%2016.png)

![image.png](image%2017.png)

- N-buffers 에서 생산자 (Producer)의 동작 과정

생산자의 목표는 "빈 칸을 찾아 데이터를 채워 넣는 것"입니다.

1. **`P(empty)` (empty 확보)**
    - 현재 버퍼에 비어있는 칸이 있는지 확인합니다.
    - `empty` 값이 0보다 크면 1 감소시키고 다음 단계로 가고, 0이면 소비자가 데이터를 가져가서 빈 칸을 만들 때까지 대기합니다.
2. **`P(mutex)` (임계 영역 진입)**
    - 공유 버퍼에 접근하기 전 자물쇠를 겁니다. 다른 생산자나 소비자가 동시에 버퍼를 건드리지 못하게 상호 배제를 보장합니다.
3. **`Critical Section` (데이터 저장)**
    - 실제로 버퍼의 특정 인덱스에 데이터를 씁니다. (빨간색 영역)
4. **`V(mutex)` (자물쇠 해제)**
    - 데이터 쓰기가 끝났으므로 자물쇠를 풀고 나옵니다.
5. **`V(full)` (full 개수 증가)**
    - "이제 소비자가 가져갈 데이터가 하나 생겼다"고 신호를 보냅니다. `full` 값을 1 증가시킵니다.

---

- N-buffers 에서 소비자 (Consumer)의 동작 과정

소비자의 목표는 "찬 칸에서 데이터를 꺼내 가는 것"입니다.

1. **`P(full)` (데이터 유무 확인)**
    - 버퍼에 가져갈 데이터가 있는지 확인합니다.
    - `full` 값이 0이면 생산자가 데이터를 넣을 때까지 대기하고, 0보다 크면 1 감소시키고 진행합니다.
2. **`P(mutex)` (임계 영역 진입)**
    - 공유 버퍼에 접근하기 전 자물쇠를 겁니다.
3. **`Critical Section` (데이터 추출)**
    - 버퍼에서 데이터를 읽어옵니다. (빨간색 영역)
4. **`V(mutex)` (자물쇠 해제)**
    - 작업이 끝났으므로 자물쇠를 풉니다.
5. **`V(empty)` (empty 개수 증가)**
    - "내가 데이터를 하나 빼갔으니 빈 칸이 하나 생겼다"고 생산자에게 알립니다. `empty` 값을 1 증가시킵니다.

### 4.4 Reader-Writer 문제

![image.png](image%2018.png)

- **목적:** Reader들은 동시에 읽을 수 있지만, Writer는 단독으로만 접근해야 함.
- **설정:** * `wmutex = 1` (Writer 간 또는 Reader-Writer 간 상호배제)
    - `rmutex = 1` (Reader 수 업데이트 시 보호)
    - `readCount = 0` (현재 읽고 있는 Reader 수)
- **과정 (Reader):**
    1. `P(rmutex); readCount++;`
    2. `if (readCount == 1) P(wmutex);` (첫 번째 독자는 Writer를 막음)
    3. `V(rmutex); 읽기 수행;`
    4. `P(rmutex); readCount--;`
    5. `if (readCount == 0) V(wmutex);` (마지막 독자가 Writer에게 길을 열어줌)
    6. `V(rmutex);`
