# 1주차 — 네트워크 개요 & OSI 계층

---

## 1️⃣ 네트워크란 무엇인가

네트워크(Network)란  
두 개 이상의 장치가 데이터를 주고받을 수 있도록 연결된 구조를 의미한다.

### 예시
- 스마트폰 ↔ 서버 (카카오톡, 인스타그램)
- 노트북 ↔ 노트북 (파일 전송)
- 브라우저 ↔ 웹 서버 (HTTP 통신)

 **핵심 포인트**
- 단순한 연결 ❌  
- 정해진 규칙(프로토콜)에 따라 통신 ⭕

프로토콜 : 컴퓨터나 기기들이 데이터를 주고받기 위해 미리 정해놓은 약속(데이터의 형식, 순서, 오류 처리 방식 등 정의)
---

## 2️⃣ 통신의 기본 요소

네트워크 통신에는 항상 아래 **4가지 요소**가 필요하다.

| 요소 | 설명 |
|---|---|
| 송신자 (Sender) | 데이터를 보내는 쪽 |
| 수신자 (Receiver) | 데이터를 받는 쪽 |
| 메시지 (Message) | 전달하려는 데이터 |
| 프로토콜 (Protocol) | 통신 규칙 |

### 예시 — 브라우저에서 네이버 접속
- 송신자: 내 브라우저  
- 수신자: 네이버 서버  
- 메시지: “이 페이지 주세요”  
- 프로토콜: HTTP  

---

## 3️⃣ 프로토콜이 필요한 이유

 **IF 규칙이 없다면**
- 데이터 형식이 제각각
- 전송 순서 엉킴
- 오류 처리 불가


 **프로토콜이 정하는 것**
- 데이터 형식
- 전송 순서
- 오류 발생 시 처리 방법

---

## 4️⃣ 계층화의 이유 (왜 나눴나?)

네트워크는 매우 복잡한 시스템이다.  
→ 한 덩어리로 설계하면 유지보수가 불가능하다.

### 계층화의 장점

#### 1️⃣ 문제 분리
- 특정 계층만 수정 가능
- 전체 시스템에 미치는 영향 최소화

#### 2️⃣ 표준화
- 장비나 운영체제가 달라도 통신 가능

#### 3️⃣ 개발 & 디버깅 용이
- “어느 계층에서 문제인가?”를 빠르게 파악 가능
 
> “네트워크 계층화는 복잡한 통신 과정을 역할별로 나눠 관리하기 위한 구조다.”

---

## 5️⃣ 캡슐화 (Encapsulation)

![캡슐화](./참고자료/캡슐화.png)

계층화된 네트워크에서  
데이터는 한 번에 전달되지 않고  
**상위 계층에서 하위 계층으로 내려가며 각 계층의 정보가 덧붙여진다.**

이를 캡슐화(Encapsulation)라고 한다.

 **캡슐화 한 줄 정의**  
> 상위 계층의 데이터에 하위 계층이 자신의 제어 정보를 덧붙여 전송하는 과정

---

### 캡슐화 흐름 예시 (송신 측)

브라우저에서 웹 요청을 보낼 때

- **Application**  
  - “이 페이지 주세요” (HTTP 데이터)

- **Transport**  
  - TCP 헤더 추가  
  - (포트 번호, 순서, 오류 제어 정보)

- **Network**  
  - IP 헤더 추가  
  - (출발지 IP, 목적지 IP)

- **Data Link**  
  - 프레임 헤더 추가  
  - (MAC 주소)

- **Physical**  
  - 전기 신호로 변환되어 전송

**핵심 포인트**
- 각 계층은 **자기 역할만 수행**
- 하위 계층은 상위 계층의 내용을 해석하지 않음
- 계층 간 **독립성 보장**
- 문제 발생 시 **원인 계층 추적 가능**

 **캡슐화의 의미**  
> “캡슐화는 네트워크 계층들이 서로 독립적으로 동작하면서도  
> 하나의 통신을 완성할 수 있게 해주는 핵심 메커니즘이다.”

---

## 6️⃣ OSI 7계층 한 줄 요약

| 계층 | 이름 | 한 줄 설명 |
|---|---|---|
| 7 | Application | 사용자와 가장 가까운 계층 |
| 6 | Presentation | 데이터 형식 변환, 암호화 |
| 5 | Session | 연결 관리 |
| 4 | Transport | 신뢰성, 흐름 제어 |
| 3 | Network | 경로 선택 |
| 2 | Data Link | 같은 네트워크 내 전송 |
| 1 | Physical | 전기 신호 |

---

## 7️⃣ TCP/IP 모델 개요

OSI 모델은 **개념적 모델**이고,  
실제 인터넷에서는 **TCP/IP 모델**을 사용한다.

### TCP/IP vs OSI 대응 관계

| TCP/IP | OSI |
|---|---|
| Application | 7, 6, 5 |
| Transport | 4 |
| Internet | 3 |
| Network Access | 2, 1 |

 **TCP/IP 모델 특징**
- 실무 중심
- 인터넷 표준

---

## 8️⃣ OSI vs TCP/IP 차이

![TCP/IP, OSI 비교](./참고자료/TCPOSI.png)

| OSI | TCP/IP |
|---|---|
| 이론적 모델 | 실제 사용 |
| 7계층 | 4계층 |
| 교육용 | 실무용 |
| ISO 제안 | DARPA 제안 |

## 9️⃣ 전송 방식의 종류  

![캐스트종류](./참고자료/전송방식.jpg)

(유니캐스트 / 브로드캐스트 / 멀티캐스트 / 애니캐스트)

네트워크에서 데이터는  
누구에게 전달하느냐에 따라 전송 방식이 구분된다.

---

### 1️⃣ 유니캐스트 (Unicast)

- **1 : 1 통신**
- 하나의 송신자가 하나의 수신자에게 데이터 전송

 **예시**
- 웹 서버 ↔ 브라우저
- 파일 다운로드
- 일반적인 TCP 통신

 **특징**
- 가장 기본적인 통신 방식
- 대상이 명확
- 트래픽 제어 용이

---

### 2️⃣ 브로드캐스트 (Broadcast)

- **1 : N 통신 (같은 네트워크 내 모든 노드)**
- 특정 네트워크에 연결된 모든 장치에게 데이터 전송

 **예시**
- ARP 요청
- DHCP Discover

 **특징**
- 네트워크 전체에 패킷 전송
- 불필요한 트래픽 발생 가능
- 라우터를 넘어가지 않음

---

### 3️⃣ 멀티캐스트 (Multicast)

- **1 : 그룹 통신**
- 특정 그룹에 가입한 노드들만 데이터 수신

 **예시**
- IPTV
- 실시간 스트리밍
- 화상 회의

 **특징**
- 브로드캐스트보다 효율적
- 그룹 관리 필요
- UDP 기반 사용이 일반적

---

### 4️⃣ 애니캐스트 (Anycast)

- **1 : 1 (가장 가까운 노드)**
- 동일한 IP를 가진 여러 서버 중  
  **네트워크 상 가장 가까운 서버로 전송**

 **예시**
- DNS 서버
- CDN

 **특징**
- 응답 속도 향상
- 부하 분산 효과
- 라우팅에 의해 목적지 결정

---

 **전송 방식 요약**

| 방식 | 대상 |
|---|---|
| Unicast | 하나의 특정 수신자 |
| Broadcast | 같은 네트워크의 모든 노드 |
| Multicast | 특정 그룹 |
| Anycast | 가장 가까운 노드 |

## 🔟 네트워크 토폴로지 (Network Topology)

### Network Topology란?

컴퓨터 네트워크의 구성 요소들을  
**물리적 또는 논리적으로 어떻게 연결했는지**를 나타낸 구조를 의미한다.

연결 방식에 따라 네트워크의 성능, 안정성, 확장성이 달라진다.

---

### 1️⃣ Bus 토폴로지

![버스형](./참고자료/버스형.png)

하나의 단일 회선(버스)에 여러 노드가  
T자 형태로 연결된 구조

- 회선 끝에는 종단기(Terminator)를 설치하여 신호 반사 방지
- 한 시점에 하나의 노드만 회선 사용

 **전송 방식**
- CSMA/CD
- Token Passing

 **장점**
- 구조가 단순
- 노드 추가 및 삭제 용이

 **단점**
- 대역폭 공유 → 노드 증가 시 성능 저하
- 버스 장애 시 전체 네트워크 영향

---

### 2️⃣ Star 토폴로지

![성형](./참고자료/성형.png)

중앙 노드(Hub, Switch)를 중심으로  
각 노드가 **Point-to-Point** 방식으로 연결된 구조

 **예시**
- Ethernet 네트워크

 **장점**
- 확장성 우수
- 개별 노드 장애가 전체 네트워크에 영향 없음

 **단점**
- 중앙 노드 장애 시 전체 네트워크 마비

---

### 3️⃣ Ring 토폴로지

![링형](./참고자료/링형.png)

노드들이 **원형 구조**로 연결되어  
앞 노드에서 받은 데이터를 다음 노드로 전달

 **전송 방식**
- Token Passing

 **장점**
- 충돌 방지
- 설치 비용 비교적 저렴

 **단점**
- 하나의 노드 장애가 전체 네트워크에 영향
- 노드 수 증가 시 성능 저하
- 대규모 네트워크에 부적합

---

### 4️⃣ Mesh 토폴로지

![메쉬형](./참고자료/메쉬형.png)

모든 노드가 **그물 형태로 연결**된 구조

 **사용 사례**
- 백본 네트워크
- 라우터 간 연결

 **장점**
- 높은 가용성
- 특정 노드 장애 시에도 다른 경로로 통신 가능

 **단점**
- 설치 및 관리 복잡
- 비용 높음

---

### 5️⃣ Tree 토폴로지

![트리형](./참고자료/트리형.png)

계층적 구조로 노드들이 연결된 형태  
(Star + Bus 구조 혼합)

 **장점**
- 네트워크 관리 용이
- 구조적 분류 가능

 **단점**
- 상위 노드 트래픽 집중
- 장애 발생 시 영향 범위 큼

---

 **토폴로지 요약**

| 토폴로지 | 특징 |
|---|---|
| Bus | 단일 회선 공유 |
| Star | 중앙 집중 |
| Ring | 원형 연결 |
| Mesh | 다중 경로 |
| Tree | 계층 구조 |
